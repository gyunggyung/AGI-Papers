# 똑똑한 선생님한테 배웠는데 왜 성적은 떨어질까?

최근 작은 추론 모델(Student)을 똑똑하게 만들기 위해 가장 많이 쓰는 방법은 Claude나 GPT같은 거대 모델(Teacher)이 만든 고품질 데이터를 그대로 주입하는 SFT(지도 미세 조정)입니다.

그런데 Qwen3-8B 같은 추론 모델에 이 짓을 하면 코딩 테스트 성적이 무려 10%나 폭락하는 기현상이 발생합니다. 더 좋은 데이터를 먹였는데 왜 바보가 될까요?

TESSY(Teacher-Student Cooperation Framework) 논문은 이 역설의 원인을 '스타일 충돌'에서 찾고, 기발한 해결책을 제시합니다.

### 1. 파이썬 개발자와 C++ 장인

파이썬만 5년 파서 자신만의 코딩 스타일이 확고한 개발자(학생 모델)가 있습니다. 이 사람에게 세계 최고의 C++ 장인(교사 모델)이 고급 알고리즘을 가르칩니다.

이때 장인이 핵심 알고리즘 논리만 알려주는 걸 넘어, 변수명 짓는 법이나 중괄호 위치 같은 'C++식 코딩 스타일'까지 강제로 똑같이 따라 쓰라고 압박하면 어떻게 될까요? 

파이썬 개발자는 문법 오류를 연발하며 원래 할 줄 알던 기본기마저 잃어버리게 될 겁니다. 이것이 바로 파괴적 망각(Catastrophic Forgetting)입니다.

### 2. 고품질 데이터의 역설 (Teacher-Only의 실패)

추론 모델들은 "Okay, let's see", "Wait" 같은 자신만의 고유한 말투와 생각 전개 방식(Style)을 이미 굳힌 상태입니다.

여기에 출처와 억양이 완전히 다른 교사 모델(GPT-OSS-120B)의 데이터를 통째로 덮어씌우면 분포 충돌이 일어납니다. 

실제로 교사 모델이 만든 데이터만으로 학습시켰을 때, Qwen3-8B는 OJBench에서 10.02%, LiveCodeBench-V5에서 13.77%나 성능이 하락했습니다.

### 3. 해결책: 논리는 선생님이, 말투는 학생이 (TESSY)

논문은 TESSY라는 프레임워크를 도입합니다. 데이터를 만들 때 역할을 철저히 나눈 것입니다.

 - **교사 모델**: 문제 해결에 필요한 '핵심 추론 토큰(Capability)'만 생성합니다.
 - **학생 모델**: 추론 사이사이를 잇는 '스타일 토큰(Style)'을 생성합니다.

이렇게 교사의 뛰어난 추론 논리를 챙기면서도 학생의 원래 스타일을 유지한 협업 데이터를 학습시키자, 성능 하락은커녕 오히려 11.25%, 6.68%씩 성능이 수직 상승했습니다.

### 4. 스타일 일치 vs 절대적 품질, 결정적 실험

논문 부록에는 이 주장에 쐐기를 박는 충격적인 실험이 있습니다. 문제를 다 풀고 '최종 정답'을 누가 쓰게 할 것인가?

 - 정답률이 더 높은 교사 모델이 작성하게 함 -> 성능 13% 폭락
 - 정답률은 좀 낮아도 학생 모델이 작성하게 함 -> 성능 상승

데이터의 절대적인 품질이 조금 떨어지더라도, 학생 모델의 스타일을 깨지 않는 것이 SFT에서 훨씬 더 중요하다는 것을 증명한 셈입니다. (이는 사전학습만 거친 Base 모델에서도 동일하게 나타나는 현상입니다.)

### 💡 마치며: 작고 효율적인 모델을 만드는 우리의 과제

우리는 한정된 자원 안에서 작은 모델(sLM)을 한계까지 끌어올리기 위해 거대 모델의 지식을 증류(Distillation)하는 데 집중해왔습니다.

하지만 이 논문은 단순히 "좋은 데이터를 많이 때려 넣는다"는 접근이 능사가 아님을 보여줍니다. 작은 모델이 원래 가지고 있던 데이터 분포와 스타일을 존중하지 않으면 아무리 좋은 지식도 독이 됩니다.

자원이 부족한 환경에서 로컬 추론을 위한 최적의 모델을 깎아야 하는 우리에게, '무엇을 가르칠 것인가' 못지않게 '어떤 그릇(스타일)에 담아 먹일 것인가'가 중요하다는 귀중한 힌트를 줍니다.
