# LFM2-350M-ToolLLaMA: 14달러로 GPT-5-Nano를 이기다

> **14달러로 Tool Calling(ToolBench)에서 OpenAI GPT-5-Nano보다 6.6배 이상 좋은 350M 모델을 만들었습니다.** AWS의 "Small Language Models for Efficient Agentic Tool Calling"을 직접 재현한 프로젝트입니다.

동급 크기 대비 현존 최고인 Liquid AI의 LFM2-350M을 ToolBench 18만 7천 개 데이터로 튜닝했습니다. H100 한 대, 학습 4시간, 총 14달러. 그 결과가 **LFM2-350M-ToolLLaMA**입니다.

단순히 사용할 도구를 고르는 게 아닙니다. 사고의 흐름(ReAct 형식 Chain of Thought)이 담겨 있습니다. "노래 찾아줘"라고 했을 때 Shazam API가 에러를 뱉으면, 포기하지 않고 "API가 막혔네? 그럼 구글 검색을 해봐야지"라고 스스로 판단하고 우회합니다.

## 🎯 결과

| Metric | LFM2-350M-ToolLLaMA | GPT-5-Nano |
|---|---|---|
| **Format Accuracy** | **99.9%** | 78.6% |
| **Action Match Rate** | **56.3%** | 8.5% |
| 도구 2~3개 명시 시 | **100%** (4/4) | - |
| 총 비용 | **$13.80** | - |

## 📊 무엇을 증명했나

Liquid AI의 공식 LFM2-1.2B-Tool 모델이 ToolBench 평가에서 0.0%를 기록했습니다. 3.5배 더 큰 모델인데 말이죠. 이유는 학습 데이터 형식 차이. Pythonic 함수 호출 형식으로 학습된 모델에게 ReAct 형식을 요구하니 당연히 0%입니다.

**모델 크기보다 학습 데이터 형식 정렬이 더 중요합니다.** "파라미터 희석" 문제를 직접 확인한 셈입니다.

## 🧠 실전에서 어떻게 쓸까

350M 한 대로 만 개의 도구를 다 알 필요는 없습니다. 앞단에 Fastino Labs의 GLiNER2 같은 작은 라우터(인코더 모델)를 두고, 쿼리를 분류해서 해당 도구 2~3개만 프롬프트에 넣어주면 됩니다. 이 구조에서 350M은 **100%**를 찍었습니다.

**작은 모델 + 스마트 라우팅 = 큰 모델보다 빠르고 저렴하고 정확한 시스템.**

## 🛠️ 학습 세부사항

| Item | Detail |
|---|---|
| Base | Liquid AI LFM2-350M (LNN 기반 하이브리드 모델) |
| 데이터 | ToolBench 187,494개 (LFM ChatML 형식 변환) |
| 방식 | Full SFT, TRL SFTTrainer, 1 Epoch |
| 환경 | VESSL AI H100 80GB × 1 |
| 시간 | 4시간 4분 14초 (5,891 steps) |
| 최대 VRAM | 53.73GB / 80GB |
| Mean Token Accuracy | 95.64% |
| 최종 Loss | 0.1668 |

## 📉 솔직한 한계

56.3%는 엄격한 문자열 매칭 기준입니다. "latest_coupons"를 "trending_coupons"로 예측하면 의미적으로 맞아도 실패 처리됩니다. 논문의 77.55%는 ChatGPT가 판정하는 Pass Rate이라 직접 비교가 불가능합니다.

그리고 ToolLLaMA 데이터셋에 오버피팅 돼 있을지도 모르겠습니다. 학습 최적화 여지도 남아 있고, RL 작업도 아직 진행하지 않았습니다.

## 🚀 다음 스텝

이 모델을 Tiny-MoA 프로젝트에 통합할 겁니다. Brain(LFM2.5-1.2B)이 의도를 파악하고, 이 350M 모델이 실제 도구를 호출하는 구조입니다. 독점 API 모델 없이 로컬 노트북 CPU에서 돌아가는 에이전트 시스템에 발전된 Tool Use까지 붙이는 것이 최종 목표입니다.
