# PivotRL: 아는 문제는 안 풉니다

SFT보다 4.17%p 더 높은 성능, RL보다 5.5배 더 빠른 훈련 시간, 타 도메인 지식 망각 0%. Nvidia Nemotron-3-Super-120B-A12B의 비장의 카드 PivotRL. "아는 문제는 안 풉니다."

LLM을 스스로 계획하고 행동하는 에이전트(Agent)로 훈련시킬 때 우리는 깊은 딜레마에 빠집니다. 

단순히 전문가나 Claude 같은 거대 모델을 따라 하는 지도학습(SFT)은 효율적이지만, 모델이 기존에 갖고 있던 소중한 타 도메인 지식(수학, 코딩 등)을 까먹는 치명적인 '망각 현상'이 발생합니다. 

반대로, 망각을 막고 환경에 적응시키기 위해 궤적 전체를 처음부터 끝까지 수십 번 시뮬레이션하는 종단간 강화학습(E2E RL)을 돌리자니 어마어마한 컴퓨팅 자원이 소모됩니다.

최근 NVIDIA는 이 딜레마를 뚫어낼 기발한 타협점, 'PivotRL'을 발표했습니다. 이 프레임워크는 기존 전문가 시연 궤적에서 가장 중요한 중간 단계, 즉 '피벗(Pivot)'만을 찾아내 집중 훈련합니다.

## 1. 퀵 정렬(Quick Sort)의 지혜: 최적의 피벗 찾기

우리가 알고리즘 시간에 배운 퀵 정렬의 성능은 배열을 정확히 반으로 가르는 '피벗'을 어떻게 고르느냐에 완벽히 달려 있습니다. 만약 무작위로 피벗을 고르거나 이미 정렬된 값의 끝부분을 고르면, 무의미한 연산만 반복하며 성능이 곤두박질칩니다.

PivotRL의 핵심도 이와 완벽하게 일치합니다. 전체 작업 궤적 중 무작위 턴을 고르면, 71%의 확률로 모델이 이미 완벽히 맞추거나 100% 틀려버리는 무의미한 구간에 걸립니다. 

모델의 행동 결과가 100% 똑같으면, 보상의 분산(어드밴티지)이 없어 가중치를 업데이트할 학습 신호가 완전히 죽어버립니다.

그래서 PivotRL은 두 가지 조건으로 가장 가성비 좋은 오답노트를 만듭니다.

 - 헷갈리는 구간인가? (보상 분산 > 0): 시도할 때마다 성공과 실패가 오락가락하는 곳.

 - 아직 어려운 구간인가? (보상 평균 < 기준치): 여전히 오답률이 높아 배울 점이 남아있는 곳.

마치 100줄짜리 코드를 디버깅할 때, 1번 줄부터 무식하게 처음부터 끝까지 돌리는(E2E RL) 대신 버그가 터질락 말락 하는 아슬아슬한 경계선(피벗)에 브레이크포인트를 걸고 거기서부터만 로컬 테스트를 돌리는 극강의 효율화 전략입니다.

## 2. 숫자가 증명하는 압도적 가성비

오직 이 '피벗' 지점에만 모델을 떨궈놓고 짧은 롤아웃을 실행한 결과는 놀랍습니다. (동일한 훈련 데이터 사용 기준)

 - 도메인 내(In-Domain) 성능 폭발: 베이스 모델 대비 평균 +14.11%p 상승하며, 일반 SFT(+9.94%p)를 가볍게 눌렀습니다.

 - 망각의 완벽한 방어 (OOD 보존): 일반 SFT는 터미널 환경 훈련 후 수학 점수(AIME)가 -64.48%p나 폭락하는 대참사를 겪었지만, PivotRL은 타 벤치마크 성능 변화를 평균 +0.21%p로 유지하며 망각을 원천 차단했습니다.

 - 연산량 4배 다이어트: 복잡한 소프트웨어 엔지니어링 벤치마크(SWE-Bench)에서 E2E RL과 동일한 정확도에 도달하는 데 누적 롤아웃 턴 수를 약 4배 줄이고, 실제 훈련 시간(Wall-clock time)을 5.5배나 단축시켰습니다.

## 3. 정답과 달라도 괜찮아: 깐깐한 편집장 해고하기

또한 PivotRL은 무조건 정답 텍스트와 100% 똑같아야만 점수를 주는 깐깐한 SFT의 방식을 버렸습니다. 에이전트의 세상에서는 코드를 조금 다르게 짜든, 다른 검색 쿼리를 쓰든 결과만 같으면 그만입니다.

따라서 도메인 특화 '검증기(Verifier)'를 도입해 텍스트가 달라도 기능적으로 동등하면 보상을 주는 '기능적 동등성 보상'을 채택했습니다. 

텍스트 암기가 아닌, 환경에 유연하게 대처하는 방법을 가르친 것입니다. (논문에 따르면, 이 기능적 보상 덕분에 정답을 내는 행동에만 확률 질량을 더해주고 다른 훈련과 무관한 행동들의 순위는 그대로 보존되어, OOD 망각을 완벽히 방어할 수 있었다고 수학적으로 증명했습니다.)

## 4. 진짜 현실: 껍데기 검증기의 한계와 나아갈 길

물론 당장 내일 이 기술을 현업에 무지성 복붙하기엔 한계가 명확합니다. 

논문이 말하는 '기능적 보상 검증기'의 실체를 뜯어보면, 복잡한 코딩 환경에서 코드 패치의 질을 평가하는 것이 아니라, 고작 '검색', '열기' 같은 '도구의 종류(Tool name)'를 올바르게 골랐는지 정도만 확인하는 얕은 룰 기반의 꼼수에 불과합니다. 

진짜 시스템 제어 에이전트를 만들 때 이런 하드코딩된 규칙은 금방 바닥을 드러냅니다.

💡 마치며

그럼에도 불구하고 이 논문이 증명한 철학은 분명합니다. 컴퓨팅 자원이 부족한 상황에서 무작정 SFT를 반복하거나 뼈 빠지는 E2E RL 롤아웃을 돌릴 필요는 없습니다. 

무거운 강화학습 엔진(GRPO)에 밥을 주기 전에, "우리 데이터에서 진짜로 훈련할 가치가 있는 텐션 높은 피벗(Pivot) 구간은 어디인가?"를 필터링하는 전처리 과정만으로도 모델의 지능과 효율을 공짜로 끌어올릴 수 있습니다.

향후 거친 검증기 룰을 RLVR(검증 가능한 보상에 기반한 강화학습) 기술로 대체한다면, 적은 연산량으로도 스스로 진화하는 에이전트의 무한한 가능성을 열 수 있을 것입니다.
