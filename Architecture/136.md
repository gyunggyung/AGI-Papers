title: "100만 토큰의 한계를 넘으려는 시도: BLOCKSEARCH와 HiLS-Attention"
---

최근 1M 토큰 컨텍스트 길이를 지원한다는 LLM이 많아졌습니다. 문서 수만 장을 한 번에 넣고 검색할 수 있다고 하죠. 하지만 실제 테스트해 보면, 문서가 1만 장일 때 일반 LLM의 검색 정답률은 0.2%로 추락합니다. 

모델은 정답을 알지만, 9,999개의 오답이 내뿜는 노이즈가 소프트맥스(Softmax) 확률을 오염시키는 어텐션 희석(Attention Dilution) 현상 때문입니다.

만약 성능 문제를 해결하더라도, 물리적인 장벽이 남습니다. 풀 어텐션 방식을 사용해 100만 토큰의 원본을 모두 연산하고 KV 캐시로 메모리에 올려두면, VRAM 소모와 지연 시간(Latency)이 폭발하여 사실상 서비스가 불가능해집니다.

이 성능과 효율성이라는 두 마리 토끼를 잡기 위해, 0.6B, 345M 급의 작은 모델로 100만 토큰의 한계를 넘으려는 두 논문을 소개합니다. (서로 해결하려는 문제는 다르니 자세한 내용은 각 논문을 봐보시길 바랍니다)

1. 압축은 해악이다, 원본을 유지하되 공식을 고치자: BLOCKSEARCH (Qwen3-0.6B 기반)

버클리 연구팀은 문서를 요약하거나 임베딩(점)으로 압축하면 정밀한 로직이나 철자 정보가 뭉개진다고 보았습니다. 그래서 100만 토큰 원본을 KV 캐시에 그대로 유지하되, 연산 효율을 챙기는 방식을 택합니다.

 - 수식 교정과 내부 라우팅: 오답이 정답의 확률을 갉아먹지 못하도록 어텐션 점수 차이를 강제로 벌립니다(SSMax). 

동시에 연산 폭발을 막기 위해 28개 층 중 16층에서 상위 256개 문서만 남기고 나머지는 잘라내어 상위 레이어의 연산량을 크게 줄입니다 (Routing).

 - 결과: 정보 손실이 없어 복잡한 규칙 매칭(LIMIT) 테스트에서 기존 임베딩 기반 검색기(3.5%) 성능을 BLOCKSEARCH(14.9%)로 가볍게 누릅니다. 거기다 MSA-4B (7배 큰 경쟁 모델)도 특정 과제에서는 성능으로 압도합니다.

 - 청구서와 타협점: 상위 레이어에서 연산을 쳐내더라도, 하위 레이어에는 100만 토큰의 무거운 KV 캐시가 남아 있습니다. 문맥이 길어질수록 텍스트를 한 글자씩 생성(Decode)하는 속도가 점점 느려지는 물리적 메모리 대역폭 병목을 완전히 피할 수는 없습니다.

하지만, 문서를 뇌에 적재하는 단계(Prefill)에서 발생하는 연산 폭발을 선형 스케일로 막아낸 덕분에 H100 GPU 수십 대가 아니라 A100 GPU 노드 환경에서도 100만 토큰 스케일을 터뜨리지 않고 돌릴 수 있게 되어 기존 보다 훨씬 효율적입니다.

2. 날것을 다 쥐고 있는 건 낭비다, 구조를 혁신하자: HiLS-Attention (345M ~ Olmo3-7B 기반)

반대로 텐센트 연구팀은 100만 토큰의 KV 캐시를 모두 들고 있는 것 자체가 하드웨어 낭비라고 지적하며 아키텍처를 뜯어고쳤습니다.

 - 계층형 요약과 O(1) 디코딩: 각 문서 블록에 랜드마크 토큰을 심어 LogSumExp 선형화 공식(1차 테일러 전개)을 활용해 정교한 요약본을 만듭니다. 

질문이 들어오면 이 요약본 32개(2,048 토큰)만 골라 연산합니다. 흥미로운 점은 이 요약본을 고르는 과정이 LLM의 다음 단어 예측(LM Loss)과 직접 연결되어 엔드투엔드로 학습된다는 것입니다.

 - 압도적 효율: 512K 문맥 기준, 기존 풀 어텐션 대비 문서 적재(Prefill)는 13.5배, 토큰 생성(Decode)은 15.7배 빠릅니다. 문맥이 아무리 길어져도 생성 속도가 일정한 O(1)의 상수 시간을 달성하며 KV 캐시 폭발 문제를 원천 차단했습니다.

 - 강력한 성능: 8K 길이로만 학습된 기존 RoPE 모델은 16K 스케일을 넘어가자마자 정확도 0%로 전멸합니다. 하지만 HiLS-Attention (345M 모델)은 8K로만 학습했음에도, 무려 512배가 넘는 4M 스케일에서 90% 이상의 정답률을 유지하며 완벽한 외삽 능력을 증명했습니다. 

거기다 256K로 CPT 학습 시 변수 추적(VT) 과제에서 풀 어텐션 대비 성능 50% 향상을 보입니다.

 - 청구서와 타협점: 기존 오픈소스 인프라에 바로 올릴 수 없습니다. 모델에 특수 구조를 넣고, 인접 쿼리들을 묶어 처리하는 전용 하드웨어 커널을 짜야 하며, SGLang 같은 추론 엔진 내부 소스코드까지 개조해야 하는 인프라 파편화 비용이 발생합니다.

다행인 것은 텐센트 연구팀이 공식 깃허브(GitHub)에 이 골치 아픈 전용 커널과 SGLang 엔진 수정 코드를 모두 오픈소스로 공개해 두었다는 점입니다. 즉, 개발자가 밑바닥부터 인프라를 다시 짤 필요 없이 공개된 코드를 바로 가져다 테스트하고 도입할 수 있습니다.

마치며
