# 32B가 480B 모델을 이기는 데이터 엔지니어링

> **최근 Claude Code나 Codex CLI처럼 터미널 환경에서 알아서 코딩하고 디버깅하는 에이전트들이 쏟아지고 있습니다.** 그런데 이 똑똑한 녀석들을 대체 '어떤 데이터'로 학습시켰는지는 다들 꼭꼭 숨겨두고 영업 비밀로 취급하고 있죠. 이 침묵을 깨고 NVIDIA가 터미널 에이전트 훈련의 비밀 레시피를 전격 공개했습니다. 논문 "On Data Engineering for Scaling LLM Terminal Capabilities"입니다.

## 1. 문제점: 터미널 데이터는 노가다로 못 만듭니다

사람이 터미널에서 타자 치고 디버깅하는 걸 일일이 기록할 수도 없고, AI를 시켜 합성 데이터를 만들려니 매번 새로운 도커(Docker) 환경을 띄우고 다중 턴(multi-turn) 상호작용을 거쳐야 해서 시간과 비용이 미친 듯이 깨집니다. 

터미널 능력을 키우고 싶어도 데이터 자체가 턱없이 부족했던 이유입니다.

## 2. NVIDIA의 해결책: 재활용과 9개의 도커 공장

NVIDIA는 이 병목을 'Terminal-Task-Gen'이라는 가볍고 강력한 파이프라인으로 영리하게 뚫었습니다.

- **기존 문제집 재활용 (Dataset Adapters)**: 이미 존재하는 고품질의 수학, 코딩, 소프트웨어 엔지니어링 문제를 터미널 환경에 맞게 껍데기만 바꿨습니다.
- **9개의 공장 (Pre-built Docker)**: 합성 데이터를 만들 때 매번 새 도커를 빌드하는 대신, 데이터 과학, 보안 등 9개 도메인별로 미리 세팅된 도커 이미지 9개만 돌려쓰며 데이터를 찍어냈습니다. 쓸데없는 자원 낭비를 확 줄인 것이죠.

## 3. 결과: 골리앗을 쓰러뜨린 다윗

이렇게 약 26만 개의 고품질 터미널 데이터를 모아 오직 SFT(지도 학습)로만 모델을 훈련시켰습니다. 

파라미터가 32B에 불과한 Nemotron-Terminal-32B 모델이 Terminal-Bench 2.0에서 27.4점을 기록하며, 무려 15배나 거대한 480B(A35B) 크기의 Qwen3-Coder(23.9점)를 가볍게 압도했습니다. 

모델의 크기보다 '데이터의 질과 궤적(Trajectory)'이 압도적으로 중요하다는 걸 증명한 셈입니다.

## 💡 마치며: 결국 승부는 '자체 검증'과 '강화학습'

논문의 저자들도 결론에서 "앞으로는 강화학습(RL)을 통해 모델이 스스로 터미널 실행 피드백을 보고 오류를 수정하며 장기 계획을 짜게 만드는 것이 큰 잠재력이 있다"고 말했습니다.

이제 모델의 파라미터 숫자만 무식하게 키우는 시대는 끝났습니다. 작지만 똑똑한 모델에 퀄리티 높은 데이터를 먹이고, 가혹한 샌드박스 환경(RL)에서 굴리는 자가 2026년의 승자가 될 것입니다.
