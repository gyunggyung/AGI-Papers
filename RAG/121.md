모델 성능이 안 좋았던 시대에 벡터 DB를 이용한 RAG가 유행했습니다. 

최근 LlamaIndex에서 공개한 7개의 유용한 링크들을 읽고, 저에게 질문하시는 RAG 관련 답변들을 정리해보았습니다.

1. RAG는 꼭 벡터 DB만 써야 하나요? (No)
과거에는 문맥을 이해하지 못해 텍스트 유사도만으로 문서를 뽑아야 했습니다. 하지만 지금은 LLM의 지능 자체가 매우 똑똑해졌습니다. 문장 대 문장의 조밀(Dense) 검색이 아닌, 지식 그래프(Knowledge Graph)나 하이브리드(Grep + TF-IDF) 검색처럼 데이터의 특성에 맞는 다양한 구조가 훨씬 더 뛰어난 성능을 보입니다. 무조건 벡터 DB를 띄우는 것부터 시작하는 것은 구시대적인 접근입니다.

2. 검색된 문서(Context)를 다 LLM에 넣어야 하나요? (No)
모델들이 100만 토큰을 읽을 수 있게 되었지만, 필요 없는 내용까지 다 욱여넣으면 추론 속도가 느려지고 쓸데없는 비용이 발생하며, 심지어 모델이 혼란을 느껴 환각(거짓말)을 할 확률이 높아집니다. 
리랭커(Reranker)나 컨텍스트 압축(PISCO, LimRank) 기술을 적용해, 질문과 100% 매칭되는 핵심 정보만 핀셋처럼 추려내어 던져주어야 합니다. 적은 문서로 완벽한 답을 유도하는 것이 진정한 기술입니다.

3. RAG와 SFT(파인튜닝) 중 무엇을 해야 하나요?
- SFT: 모델에게 "보고서 양식에 맞게 JSON 형태로 대답해"처럼 말투나 출력 포맷을 교육하는 과정입니다. 새로운 지식을 가르칠 수는 없습니다.
- RAG: 모델에게 매번 최신 정보(매뉴얼, 법규 등)가 적힌 가림판(컨텍스트)을 보여주며 답변하게 만드는 과정입니다. 실시간으로 변하는 최신 정보를 다룰 때는 RAG가 정답입니다.

4. 1B, 3B 급 소형 모델(sLM)로도 RAG가 가능한가요? (Yes)
최근 sLM 모델들의 지능이 급격히 상승했습니다. RAG 파이프라인에서 검색을 지시하는 도구 호출(Tool Calling)이나, 검색된 문서를 파싱하는 용도로는 1B~3B 모델로도 충분히 훌륭한 시스템을 구축할 수 있습니다. 

5. 추천하는 7가지 핵심 리소스 (LlamaIndex)
- LlamaIndex Core Concepts
- Advanced Query Pipelines
- Knowledge Graph RAG Guide
- Context Compression & Filtering
- Agentic Retrieval Architecture
- Evaluation & Observability (Ragas)
- Multi-Document Agent Workflow


마치며: 화려한 아키텍처보다 중요한 데이터 엔지니어링

많은 분들이 거창하고 화려한 RAG 프레임워크나 비싼 벡터 DB 유료 결제에 집중하지만, RAG의 본질은 결국 '질문에 맞는 정확한 텍스트 조각을 어떻게 찾아내느 하드 엔지니어링'에 있습니다. 

데이터를 이쁘게 정제하고, 현업의 업무 지식에 맞춰 파싱 규칙을 짜는 데이터 엔지니어링의 노가다가 RAG 성능의 90%를 결정합니다. 기본을 놓치지 않길 바랍니다.
